# Стратегия Тестирования

В проекте используем несколько уровней проверок. Они могут быть похожи по шагам, но отвечают на разные вопросы.

## Unit-Тесты

Unit-тесты проверяют маленькую часть логики без React-страницы и браузера.

Примеры:

- mapper превращает API-ответ в данные для таблицы;
- selector считает агрегаты по инцидентам;
- reducer правильно сохраняет состояние ошибки.

Главный вопрос: **логика сама по себе работает правильно?**

### Мини-правило Для Unit-Тестов

Unit-тест кладём рядом с проверяемой логикой: `*.test.ts` около mapper/selector/reducer/util.

Если у логики и теста получается одно имя (`aggregateCharts.ts` + `aggregateCharts.test.ts`), выносим их в папку по имени фичи/утилиты:

```text
utils/
  aggregateCharts/
    index.ts
    aggregateCharts.test.ts
```

Пишем unit, когда можно проверить результат обычным вызовом функции без React, роутера, браузера и API. Для incidents хороший пример — фильтрация дерева по критичности или агрегация данных для графиков.

Запуск:

```bash
npm test
```

E2E-spec-и из `__e2e__` не запускаются Vitest-ом, для них есть отдельные Playwright-скрипты.

## Coverage

Процент покрытия считаем через Vitest по исходникам всех приложений в `apps/*/src`.

В coverage входят unit и integration-тесты. E2E на Playwright контролирует пользовательские сценарии отдельно и в процент покрытия кода не входит.

Из процента исключены:

- `__e2e__`;
- `*.test.*` и `*.spec.*`;
- style-файлы;
- локали;
- type-only файлы;
- bootstrap `apps/*/src/app/index.tsx`.

Основные команды:

```bash
npm run test:coverage
npm run test:coverage:html
npm run test:coverage:open
npm run test:coverage:check
```

`test:coverage` показывает текстовый отчёт и сохраняет артефакты в `coverage/apps`.

`test:coverage:html` создаёт HTML-отчёт: `coverage/apps/index.html`.

`test:coverage:open` открывает последний HTML-отчёт в браузере.

`test:coverage:check` проверяет порог 80% по statements, branches, functions и lines. Пока покрытие ниже 80%, команда будет падать — это нормально для контроля в CI.

## Integration-Тесты

Integration-тесты проверяют связку нескольких частей внутри модуля: компонент, props, local state, store, hooks, провайдеры.

Пример для incidents:

```text
отрендерить IncidentsTable с mock data
кликнуть expander первой строки
проверить, что строка "Блок 1" скрылась
```

Главный вопрос: **компонент или модульная связка работает правильно в изоляции?**

Такой тест не открывает реальный роут `/incidents` и не проверяет всю страницу приложения.

## E2E-Тесты

E2E-тесты проверяют пользовательский сценарий в приложении через Playwright.

Пример для incidents:

```text
открыть /incidents
дождаться данных
кликнуть expander первой строки
проверить, что строка "Блок 1" скрылась
сравнить snapshot страницы
```

Главный вопрос: **пользовательский сценарий работает в собранном приложении?**

Здесь проверяется не только таблица, а вся связка: роут, страница, layout, API-моки, провайдеры, тема, i18n и итоговый UI в браузере.

## Storybook

Storybook нужен для разработки и визуальной документации компонентов.

Примеры:

- показать `IncidentsTable` с разными состояниями данных;
- показать empty/loading/error;
- быстро проверить внешний вид без запуска всего приложения;
- добавить interaction через `play`, если нужно проверить простой клик внутри компонента.

Главный вопрос: **как компонент выглядит и ведёт себя в конкретном состоянии?**

## Почему Integration И E2E Могут Повторять Шаги

Да, шаг может быть одинаковым:

```text
кликнуть expander первой строки
проверить, что "Блок 1" скрыт
```

Но смысл проверки разный.

Integration-тест проверяет: **сама таблица правильно обрабатывает клик?**

E2E-тест проверяет: **на реальной странице `/incidents` пользователь может выполнить этот сценарий, и вся страница после этого выглядит правильно?**

То есть повторяется действие, но не повторяется уровень риска.

## Практическое Правило

Если проверяем чистую логику — пишем unit.

Если проверяем компонент или модульную связку с mock data — пишем integration или Storybook interaction.

Если проверяем реальный пользовательский путь через роут приложения — пишем E2E.

Если нужно показать состояния UI для разработчиков и ревью — пишем Storybook story.

Для одного сценария не надо писать все виды тестов всегда. Достаточно выбрать минимальный уровень, который закрывает риск.
