# Описание Архитектуры Фронтенда SPA 2

## Для Внутреннего Рассмотрения Руководством И Подготовки К Требованиям ФСТЭК

Версия документа: 1.0

Дата: 28.05.2026

Статус: проектный архитектурный документ

Объект описания: фронтенд-шаблон SPA 2

Репозиторий: `spa2.0`

Ответственная зона: клиентская часть, UI-платформа, тестовая инфраструктура фронтенда

## 1. Назначение Документа

Настоящий документ описывает архитектуру фронтенд-шаблона SPA 2, текущие проектные решения, механизмы контроля качества, тестирования и сопровождения.

Документ подготовлен как управленческий и технический артефакт для:

- демонстрации архитектурной готовности шаблона руководству;
- передачи подходов разработки другим командам;
- подготовки доказательной базы по качеству фронтенда;
- дальнейшего развития процессов тестирования и контроля покрытия;
- предварительной подготовки к требованиям регуляторного контроля, включая практики, которые могут использоваться при подготовке к проверкам по линии ФСТЭК.

Документ не является аттестационным заключением, моделью угроз, техническим заданием на систему защиты информации или официальным комплектом документов для сертификации. Он фиксирует текущее состояние фронтенд-архитектуры и перечень мер, которые уже заложены или должны быть доведены до промышленного уровня.

## 2. Область Применения

Документ распространяется на фронтенд-часть SPA 2:

- приложения в `apps`;
- общий платформенный пакет `core`;
- Storybook-пространство `core/storybook`;
- тестовую инфраструктуру Vitest и Playwright;
- документацию в `docs`;
- правила зависимостей, сборки, качества и покрытия тестами.

Документ не описывает:

- серверную архитектуру;
- инфраструктуру эксплуатации;
- модель угроз информационной системы;
- сетевые средства защиты;
- серверную аутентификацию и авторизацию;
- криптографическую защиту;
- процессы администрирования продуктивной среды.

Эти зоны должны быть оформлены отдельными документами при подготовке полного комплекта материалов.

## 3. Краткое Резюме

Фронтенд SPA 2 построен как монорепозиторий на `npm workspaces`.

В составе шаблона выделены:

- три приложения: `app-operator`, `app-admin`, `app-expert`;
- общий платформенный пакет `@org/core`;
- единый Storybook для UI-компонентов;
- единый набор правил зависимостей;
- единая стратегия тестирования;
- настройка покрытия тестами по всем приложениям в `apps`;
- E2E-инфраструктура на Playwright;
- visual snapshots и HAR-моки для стабильных браузерных сценариев;
- документация по ключевым направлениям разработки.

Архитектура готова для демонстрации и дальнейшего использования командами как базовый шаблон. При этом текущий процент покрытия тестами ниже целевого уровня 80%; это является задачей дальнейшего развития, а не блокером для демонстрации архитектурного подхода.

## 4. Архитектурные Принципы

В проекте используются следующие принципы:

- разделение приложений и общей платформы;
- отсутствие прямой зависимости `core` от приложений;
- доменная логика хранится внутри соответствующего приложения;
- общие UI-элементы, тема, i18n, HTTP-клиент и платформенные утилиты выносятся в `@org/core`;
- публичная поверхность платформы контролируется через `core/index.ts`;
- приложения не импортируют напрямую платформенные runtime-библиотеки, если для них есть экспорт из `@org/core`;
- страницы остаются тонкими и собирают готовые модульные компоненты;
- тесты выбираются по уровню риска: unit, integration, E2E, Storybook;
- E2E не заменяет coverage, а дополняет его сценарными проверками.

## 5. Структура Репозитория

Основные директории:

```text
apps/
  app-operator/
  app-admin/
  app-expert/
core/
  components/
  i18n/
  lib/
  theme/
  storybook/
docs/
mock/

```

Назначение директорий:

- `apps` — самостоятельные фронтенд-приложения;
- `core` — общий платформенный пакет `@org/core`;
- `core/storybook` — единый Storybook для UI и демонстрационных состояний;
- `docs` — проектная документация;
- `mock` — локальный mock API для разработки и E2E;
- `scripts` — вспомогательные скрипты разработки и тестирования.

## 6. Приложения

В шаблоне предусмотрены три приложения.

`app-operator`:

- основное демонстрационное приложение;
- содержит развитый модуль `incidents`;
- использует mock API;
- содержит E2E-примеры и visual snapshots.

`app-admin`:

- приложение административного контура;
- содержит базовую структуру приложения и модуль `users`;
- готово к расширению по той же архитектурной схеме.

`app-expert`:

- приложение экспертного контура;
- содержит базовую структуру приложения и модуль `users`;
- готово к расширению по той же архитектурной схеме.

Все приложения используют общий подход к:

- Vite-конфигурации;
- React Router;
- Redux store;
- alias `@`;
- alias `@org/core`;
- сборке под Firefox 115;
- TypeScript-конфигурации.

## 7. Внутренняя Структура Приложения

Типовая структура приложения:

```text
src/
  app/
    App.tsx
    providers.tsx
    router/
    store/
  pages/
  modules/
```

`app` содержит composition root:

- подключение провайдеров;
- роутинг;
- store;
- типизированные хуки store;
- базовую инициализацию приложения.

`pages` содержит страницы. Страница не должна становиться местом хранения бизнес-логики. Её задача — собрать модульные компоненты и подключить их к маршруту.

`modules` содержит доменные модули. Внутри модуля располагаются:

- компоненты;
- hooks;
- api/thunks;
- store/slice;
- selectors;
- utils;
- types;
- locales;
- unit/integration-тесты;
- E2E-сценарии, если они относятся к доменному пользовательскому пути.

## 8. Общий Пакет `@org/core`

`@org/core` является платформенным слоем.

В него входят:

- UI-обёртки над Ant Design;
- общие UI-компоненты;
- donut chart;
- тема и theme provider;
- i18n provider и общие переводы;
- HTTP-клиент;
- API helpers;
- D3-экспорты, используемые общими графическими компонентами;
- toggle-компоненты для темы и локали.

Основное правило:

```text
apps импортируют из core,
core не импортирует из apps.
```

Публичная поверхность контролируется через:

```text
core/index.ts
```

Это снижает риск неуправляемых зависимостей и упрощает сопровождение платформы.

## 9. Управление Зависимостями

Проект использует `npm workspaces`.

Корневой `package.json` отвечает за:

- workspaces;
- общие scripts;
- devDependencies;
- overrides версий;
- команды сборки, тестирования и проверки.

`core/package.json` отвечает за runtime-библиотеки платформы:

- `antd`;
- `axios`;
- `d3`;
- `dayjs`;
- `i18next`;
- `react-i18next`;
- `styled-components`;
- другие общие библиотеки платформы.

`apps/app-*/package.json` содержит только то, что приложение использует напрямую:

- `@org/core`;
- React;
- React Router;
- React Redux;
- Redux Toolkit.

Прямые импорты `antd`, `axios`, `lodash`, `styled-components` в `apps/*/src` запрещены и контролируются ESLint-правилами.

## 10. Роутинг И Lazy-Сборка

Роутинг реализован через React Router.

Маршруты приложения расположены в:

```text
apps/app-*/src/app/router/
```

Страницы подключаются через `lazy`, что позволяет Vite выносить страницы в отдельные чанки при production-сборке.

Преимущества:

- снижение размера первого экрана;
- изоляция тяжёлых страниц;
- единый механизм расширения маршрутов;
- предсказуемая структура для всех приложений.

## 11. Управление Состоянием

Для управления состоянием используется Redux Toolkit.

Store создаётся на уровне приложения. Доменные reducer-ы подключаются через root reducer.

В модулях допускаются:

- slices;
- async thunks;
- selectors;
- typed domain utils.

Рекомендация:

- хранить в Redux состояние данных и UI-состояние, значимое для домена;
- не хранить в Redux готовые переведённые строки;
- для ошибок хранить ключ перевода, а перевод выполнять в UI.

## 12. API-Слой

Для HTTP-запросов используется общий `http` из `@org/core`.

Приложения не должны создавать собственные axios-инстансы без необходимости.

Преимущества централизованного API-слоя:

- единая точка настройки base URL;
- единая обработка ошибок;
- возможность будущего подключения interceptors;
- единый подход к логированию, токенам и обработке ошибок.

В текущем шаблоне `app-operator` использует локальный mock API через Vite proxy.

## 13. Темизация

Темизация построена на `styled-components`.

Ключевые элементы:

- `AppThemeProvider`;
- `ThemeProvider` styled-components;
- Ant Design `ConfigProvider`;
- `GlobalStyles`;
- theme tokens.

Компоненты должны использовать `styled` из `@org/core`.

Прямой импорт `styled-components` из приложений запрещён, чтобы сохранить единый платформенный слой темизации.

## 14. Локализация

Локализация построена на i18next.

Общие переводы находятся в `core`.

Доменные переводы находятся рядом с модулем.

Приложение собирает словари модулей в одном месте и передаёт их в `AppI18nProvider`.

Такой подход позволяет:

- изолировать тексты домена;
- переиспользовать общие тексты платформы;
- переключать язык без переписывания store;
- постепенно подключать новые модули.

## 15. Storybook

Storybook размещён в:

```text
core/storybook
```

Он используется для:

- визуальной документации UI-компонентов;
- демонстрации состояний компонентов;
- проверки темы и i18n в изоляции;
- согласования UI с командой, QA, аналитиками и дизайном.

Storybook не заменяет unit, integration или E2E-тесты. Он является отдельным инструментом визуального контроля и коммуникации.

## 16. Unit И Integration-Тестирование

Unit и integration-тесты выполняются через Vitest.

Unit-тесты проверяют чистую логику:

- selectors;
- reducers;
- mappers;
- utils;
- расчёты данных для UI.

Integration-тесты должны проверять связку нескольких частей модуля:

- компонент;
- props;
- hooks;
- store;
- providers;
- пользовательское действие в изоляции от реального браузерного сценария.

Текущие примеры unit-тестов:

- `filterIncidentsTree`;
- `aggregateCharts`.

Они проверяют чистую доменную логику модуля `incidents`.

## 17. Coverage

Покрытие тестами считается через Vitest.

Команды:

```bash
npm run test:coverage
npm run test:coverage:html
npm run test:coverage:open
npm run test:coverage:check
```

Coverage считается по:

```text
apps/*/src/**/*.{ts,tsx}
```

Из покрытия исключаются:

- E2E-сценарии;
- тестовые файлы;
- style-файлы;
- локали;
- type-only файлы;
- bootstrap `apps/*/src/app/index.tsx`.

Целевой уровень для контроля:

- statements: 80%;
- branches: 80%;
- functions: 80%;
- lines: 80%.

Текущий уровень покрытия ниже целевого. Это зафиксированный остаточный риск и задача развития тестовой базы.

## 18. E2E-Тестирование

E2E-тесты выполняются через Playwright.

Назначение E2E:

- проверить пользовательский сценарий в браузере;
- проверить связку роутов, страницы, провайдеров и API-моков;
- зафиксировать visual baseline через screenshots;
- подтвердить, что пользовательский путь работает целиком.

E2E не входит в процент unit/integration coverage. Он является отдельным доказательством сценарного тестирования.

Текущая E2E-инфраструктура:

- Firefox-проекты для `operator`, `admin`, `expert`;
- отдельные baseURL и порты для приложений;
- auto-start mock server;
- HAR-моки;
- JSON review-copy;
- visual snapshots;
- HTML report Playwright;
- trace при повторе теста.

Реальные E2E-примеры сейчас представлены для `app-operator`.

## 19. Моки И Детерминированность Тестов

Для E2E используется HAR как источник API-моков.

Преимущества:

- стабильные ответы API;
- независимость от случайного состояния mock database;
- воспроизводимость visual snapshots;
- возможность review API-ответов через JSON-копию.

Обновление моков и скриншотов разделено:

```bash
npm run test:e2e:update
npm run test:e2e:update:screens
```

Полный режим обновляет HAR, JSON и screenshots.

Режим screens обновляет только visual baseline.

## 21. Сборка

Сборка выполняется через Vite.

Общая команда:

```bash
npm run build
```

Порядок:

1. сборка `@org/core`;
2. сборка `app-operator`;
3. сборка `app-admin`;
4. сборка `app-expert`.

Целевой браузер сборки:

```text
Firefox 115
```

Это важно для совместимости с целевыми рабочими станциями на базе Astra Linux.

## 22. Управление Качеством Для Демонстрации И Приёмки

Для демонстрации архитектуры рекомендуется показывать:

- структуру `apps` и `core`;
- `@org/core` как платформенный слой;
- `docs/DEPENDENCIES.md`;
- `docs/TESTING_STRATEGY.md`;
- Storybook;
- unit-тесты;
- coverage HTML report;
- E2E-сценарий `app-operator /incidents`;
- visual snapshots;
- HAR + JSON review-copy.

Не рекомендуется показывать `test:coverage:check` как зелёный контроль до тех пор, пока покрытие не будет доведено до 80%. Эту команду следует демонстрировать как будущий CI-gate.

## 23. Аспекты Информационной Безопасности

В текущей фронтенд-архитектуре заложены следующие меры, полезные для дальнейшего соответствия требованиям безопасности:

- контролируемая структура зависимостей;
- запрет прямых импортов критичных платформенных библиотек в apps;
- централизованный HTTP-клиент;
- централизованная тема и i18n;
- отсутствие доменной логики в общем `core`;
- воспроизводимые тесты;
- E2E-моки без обращения к нестабильным внешним данным;
- контроль покрытия тестами;
- разделение приложений по ролям;
- документация по правилам разработки.

Ограничения:

- модель угроз не оформлена в этом документе;
- вопросы аутентификации и авторизации не реализованы как промышленный контур;
- RBAC не описан как завершённый механизм;
- защита секретов и конфигураций должна быть оформлена отдельно;
- CI/CD pipeline должен быть добавлен отдельным этапом;
- покрытие 80% ещё не достигнуто.

## 24. Остаточные Риски

На момент подготовки документа сохраняются следующие риски:

- покрытие unit/integration-тестами ниже 80%;
- E2E-покрытие есть только на примере `app-operator`;
- integration-тесты ещё не развёрнуты как отдельный слой;
- Storybook покрывает только первые компоненты;
- отсутствует CI-конфигурация в репозитории;
- требуется дальнейшее наполнение документации по безопасности;
- требуется развитие авторизации, ролей и доступа;
- при росте приложения потребуется оптимизация чанков и manual chunks.

Эти риски не блокируют демонстрацию архитектуры, но должны быть включены в план развития перед промышленным использованием.

## 25. Рекомендованный План Доведения До Промышленного Уровня

Этап 1. Зафиксировать архитектурный шаблон:

- утвердить `FRONTEND_AR.MD`;
- утвердить правила `docs/DEPENDENCIES.md`;
- утвердить стратегию тестирования;
- закрепить структуру modules/pages/app/core.

Этап 2. Расширить тестовую базу:

- добавить unit-тесты на reducers, selectors, thunks;
- добавить integration-тесты для ключевых компонентов;
- довести coverage до 80%;
- включить `test:coverage:check` в CI.

Этап 3. Расширить E2E:

- добавить сценарии для `app-admin`;
- добавить сценарии для `app-expert`;
- закрепить smoke-набор на каждый app;
- описать правила обновления HAR и screenshots.

Этап 4. Добавить CI/CD:

- lint;
- typecheck;
- unit/integration;
- coverage;
- Storybook build;
- E2E smoke;
- build всех приложений.

Этап 5. Подготовить безопасность:

- модель угроз;
- матрица ролей;
- требования к хранению токенов;
- правила работы с env;
- контроль зависимостей и уязвимостей;
- журналирование клиентских ошибок;
- требования к обновлению библиотек.

## 27. Вывод

Фронтенд-шаблон SPA 2 имеет устойчивую архитектурную основу для демонстрации и дальнейшего использования командами.

Сильные стороны:

- понятное разделение `apps` и `core`;
- единая платформа `@org/core`;
- готовая документационная база;
- настроенные тесты, coverage и E2E-инфраструктура;
- единый Storybook;
- строгий TypeScript;
- контроль импортов и зависимостей.

Ключевые условия перед промышленным применением:

- довести покрытие до целевого уровня;
- добавить CI;
- расширить E2E на все приложения;
- оформить безопасность отдельными документами;
- закрепить правила разработки как обязательные для команды.

При выполнении указанных условий архитектура может использоваться как базовый фронтенд-стандарт SPA 2 для дальнейшей разработки.
